## บทที่ 1 — ช่องเลือกไฟล์ในฟอร์ม create

### concept

ฟอร์มเดิมใช้ react-hook-form กับ type `Product` ตรง ๆ แต่ `Product` (จาก Prisma) ไม่มี field "ไฟล์" — มีแต่ `imageUrl: string | null`. ไฟล์เป็นของ**ฝั่งฟอร์มเท่านั้น** เลยประกาศ type ครอบเฉพาะหน้านี้

### code — [app/menu/create/page.tsx](../../app/menu/create/page.tsx)

```ts
type FormValues = Product & { image?: FileList };

const { register, handleSubmit } = useForm<FormValues>();
```

```tsx
<input type="file" accept="image/jpeg,image/png,image/webp" {...register("image")} />
```

จุดน่าเรียน:

- **`<input type="file">` ธรรมดา** — ไม่ต้องลง lib อัปโหลดใด ๆ, browser มีให้แล้ว
- **react-hook-form + file input**: ค่าที่ได้ตอน submit เป็น **`FileList`** ไม่ใช่ `File` เดี่ยว (input file เลือกหลายไฟล์ได้) → ต้องหยิบ `image[0]` เอง
- **`accept` = UX ไม่ใช่ security** — แค่กรองหน้าต่างเลือกไฟล์; ด่านจริงคือ allowlist ใน `saveImage()` ฝั่ง server (ยิง API ตรงข้าม `accept` ได้เสมอ)

### ⭐ takeaway

- type ของฟอร์ม ≠ type ของ DB — ขยายด้วย intersection (`Product & {...}`) เฉพาะฝั่งฟอร์ม ไม่แตะ Prisma schema

## บทที่ 2 — flow "2 ยิง": อัปโหลดก่อน แล้วค่อยสร้างสินค้า

### concept

```
[เลือกไฟล์] ──submit──▶ 1) POST /api/upload  (multipart FormData)
                            ◀── { url: "/uploads/xxx.png" }
                         2) POST /api/product (JSON + imageUrl)
```

ทำไมไม่ยัดรูปไปกับ JSON เดียว? — JSON ส่ง binary ไม่ได้ (ทวนบทที่แล้ว), และแยก `/api/upload` ไว้ใช้ซ้ำได้ทุกฟอร์ม. DB เก็บแค่ **URL** ไม่ใช่ตัวไฟล์

### code — [app/menu/create/page.tsx](../../app/menu/create/page.tsx)

```ts
async function uploadImage(file: File) {
  const form = new FormData();
  form.append("file", file);
  const res = await fetch("/api/upload", { method: "POST", body: form });
  const body = await res.json();
  if (!res.ok) throw new Error(body.message ?? "อัปโหลดรูปไม่สำเร็จ");
  return body.url as string;
}

const onSubmit = async ({ image, ...data }: FormValues) => {
  try {
    if (image?.[0]) data.imageUrl = await uploadImage(image[0]);
    const newProduct = await mutateAsync(data);
    // ...toast + invalidate + redirect เหมือนเดิม
  } catch (error) {
    toast.error(error instanceof Error ? error.message : "Failed to create product");
  }
};
```

จุดน่าเรียน:

- **`FormData` ห้ามตั้ง `Content-Type` เอง** — browser ตั้ง `multipart/form-data; boundary=...` ให้; ตั้งเองแล้ว boundary หาย = พังแบบคลาสสิก
- **`{ image, ...data }`** — แยก `FileList` ออกก่อน ไม่งั้น `JSON.stringify` จะพยายาม serialize FileList (ได้ `{}`)
- **ลำดับตายตัว**: upload → ได้ url → ใส่ `data.imageUrl` → create (ตอน create ต้องรู้ URL แล้ว)
- **`image?.[0]`** — ไม่เลือกรูปก็สร้างได้ (`imageUrl` null, schema รองรับ)
- อ่าน `body` ก่อนเช็ค `res.ok` → ข้อความ error จริงจาก server ("ไฟล์ใหญ่เกิน 5MB") ทะลุขึ้น toast ถึงผู้ใช้
- upload สำเร็จแต่ create พัง = มี**ไฟล์กำพร้า**ใน storage — ยอมรับได้ในสเกลนี้ ไม่ทำ rollback ให้ซับซ้อน

### ⭐ takeaway

- อัปโหลดรูปในฟอร์ม = 2 ยิง: multipart เอา URL → JSON create พร้อม URL
- ส่งไฟล์ = `FormData` และอย่าตั้ง `Content-Type` เอง

## บทที่ 3 — ไล่ `imageUrl` ให้ครบวงจร: API → type → การ์ด

### concept

เพิ่ม field ใน DB แล้วงานยังไม่จบ — ต้องไล่เติมทุกจุดที่ข้อมูลไหลผ่าน: create data → GET select → type ฝั่ง client → UI. ขาดจุดไหน ฝั่งหน้าเว็บได้ `undefined` เงียบ ๆ (bug หายากมาก)

### code

**เก็บ + ส่งกลับ** — [app/api/product/route.ts](../../app/api/product/route.ts) และ [app/api/menu/route.ts](../../app/api/menu/route.ts):

```ts
data: { name, price, description, imageUrl: body.imageUrl },   // POST
select: { id: true, name: true, price: true, description: true, imageUrl: true },  // GET ทั้ง 2 endpoint
```

**โชว์รูปจริง** — [components/ProductCard.tsx](../../components/ProductCard.tsx) (เดิม hardcode placeholder + filter เทา):

```tsx
{!imageUrl && <div className="absolute inset-0 z-30 aspect-video bg-black/35" />}
<img
  src={imageUrl ?? "https://avatar.vercel.sh/shadcn1"}
  alt={name}
  className={cn(
    "relative z-20 aspect-video w-full object-cover",
    !imageUrl && "brightness-60 grayscale dark:brightness-40",
  )}
/>
```

จุดน่าเรียน:

- Prisma `select` คืนเฉพาะที่ขอ — เพิ่ม field ใหม่ต้อง**ไล่เติม select ทุก endpoint** ที่ client ใช้
- **filter เทา + overlay ใส่เฉพาะ placeholder** — เงื่อนไข `!imageUrl` ผ่าน `cn()`; ไม่งั้นรูปสินค้าจริงโดนย้อมเทา
- **`alt={name}`** — รูปเนื้อหาจริงต้องมี alt สื่อความ (a11y) ไม่ใช่ text ค้างจาก template
- รับ `imageUrl` จาก client ตรง ๆ ได้เพราะ POST ล็อก `requireRole("ADMIN")` แล้ว — แต่ route นี้ยังไม่มี zod validate (หนี้เดิม)

### ⭐ takeaway

- field ใหม่ = ไล่ครบวงจร create → select → type → UI
- fallback ด้วย `??` ทำให้ของเก่าไม่ต้อง backfill

## บทที่ 4 — เตรียมขึ้น Supabase: pooled vs direct, `ENV_FILE`, scripts

### concept

แผน deploy ([deploy-plan.md](../course/deploy-plan.md)) เรียง **hardening → Supabase → Vercel → domain**. วันนี้เริ่มส่วน Supabase: DB ย้ายจาก docker local ไป managed Postgres. Supabase ให้ connection 2 เส้นและ**ห้ามใช้สลับหน้าที่**:

- **Pooled** (Supavisor, port `6543`, `?pgbouncer=true`) → `DATABASE_URL` ของ**แอป** — serverless เปิด/ปิด connection ถี่ ต้องผ่าน pooler ไม่งั้น connection เต็ม
- **Direct** (port `5432`) → `DIRECT_URL` สำหรับ**รัน migration**

### code

**prisma.config.ts** (commit `50e0638`) — สลับไฟล์ env ด้วย `ENV_FILE` + migration ใช้ direct:

```ts
config({ path: process.env.ENV_FILE ?? ".env" });   // .env local / .env.supabase prod
// ...
datasource: { url: process.env.DIRECT_URL ?? process.env.DATABASE_URL },
```

**scripts** — [package.json](../../package.json):

```jsonc
"postinstall": "prisma generate",                                  // เตรียมไว้ให้ Vercel build ผ่าน
"db:deploy": "cross-env ENV_FILE=.env.supabase prisma migrate deploy",
"db:deploy:ci": "prisma migrate deploy",
"db:studio:prod": "cross-env ENV_FILE=.env.supabase prisma studio",
```

**.env.example** — commit ได้ (`.gitignore` เติม `!.env.example` ยกเว้นจาก `.env*`) เป็นเอกสารว่า prod ต้องตั้ง env อะไรบ้าง

จุดน่าเรียน:

- repo นี้ใช้ **driver adapter** (`@prisma/adapter-pg`) — runtime อ่าน `DATABASE_URL` ใน [lib/prisma.ts](../../lib/prisma.ts), migration อ่านจาก `prisma.config.ts` → การแยก pooled/direct ทำที่ **env** ไม่ใช่ schema
- local docker ไม่มี `DIRECT_URL` → fallback `?? DATABASE_URL` ทำงาน — local กับ prod ต่างกันที่ env ไม่ใช่โค้ด
- **`cross-env`** — ไวยากรณ์ `ENV_FILE=... cmd` เป็นของ unix shell, บน Windows พัง → cross-env ทำให้ script ชุดเดียวรันได้ทุกเครื่อง
- ใช้ **`migrate deploy` ไม่ใช่ `migrate dev`** กับ Supabase — repo เปิด `partialIndexes` preview, `migrate dev` แบบ non-interactive เคยพัง
- `.gitignore` สั่งยกเว้นได้ด้วย `!pattern` (วางหลัง pattern ที่ ignore)

### ⭐ takeaway

- แอปใช้ pooled (6543), migration ใช้ direct (5432) — จำคู่นี้ให้แม่น
- แยก local/prod ด้วยไฟล์ env (`ENV_FILE`) ไม่ใช่แก้โค้ด
- `.env.example` = สัญญาของ repo ว่าต้องตั้ง env อะไร (commit ได้เพราะไม่มีค่าจริง)

## บทที่ 5 — Supabase Storage driver: เติม stub ด้วย `fetch` ล้วน

### concept

บน Vercel เขียนไฟล์ลง `public/uploads/` ไม่ได้ (serverless fs = read-only/ephemeral) → รูปต้องไป Supabase Storage. สาขา `STORAGE_DRIVER === "supabase"` ที่ค้างเป็น stub จากบทที่แล้ว วันนี้เติมจริง — และ**ไม่ต้องลง `@supabase/supabase-js`**: Storage มี REST API ตรง ๆ ใช้ `fetch` พอ

### code — [lib/storage.ts](../../lib/storage.ts)

```ts
async function saveToSupabase(filename: string, bytes: Buffer, contentType: string) {
  const url = process.env.SUPABASE_URL;
  const key = process.env.SUPABASE_SERVICE_ROLE_KEY;
  const bucket = process.env.SUPABASE_BUCKET ?? "product-images";
  if (!url || !key) throw new UploadError("Supabase env ไม่ครบ (SUPABASE_URL / SERVICE_ROLE_KEY)");

  const res = await fetch(`${url}/storage/v1/object/${bucket}/${filename}`, {
    method: "POST",
    headers: { Authorization: `Bearer ${key}`, "Content-Type": contentType },
    body: bytes,
  });
  if (!res.ok) throw new Error(`Supabase upload failed: ${res.status} ${await res.text()}`);
  return `${url}/storage/v1/object/public/${bucket}/${filename}`;   // bucket ต้อง public
}
```

จุดน่าเรียน:

- SDK ไม่ใช่ของฟรี — ถ้าใช้แค่ upload 1 ท่า, `fetch` ไป REST endpoint ตรง ๆ สั้นกว่าและไม่เพิ่ม dependency (แผนเดิมใน deploy-plan ให้ลง SDK — ของจริงไม่ต้อง)
- **`service_role` key = server-only เด็ดขาด** — มันข้าม RLS ทั้งหมด; อยู่ได้แค่ env ฝั่ง server ห้ามหลุดไป client
- เช็ค env ก่อนใช้แล้ว throw ทันที — fail fast ดีกว่าอัปโหลดเงียบ ๆ ไม่สำเร็จ
- `imageUrl` ฝั่ง prod จะเป็น **absolute URL** ของ Supabase (ไม่ใช่ `/uploads/...`) — การ์ดเรารองรับอยู่แล้วเพราะใช้ `<img>` ธรรมดา; ถ้าย้ายไป `next/image` ต้องเพิ่ม host ใน `images.remotePatterns`

### ⭐ takeaway

- serverless เขียน local disk ไม่ได้ → ไฟล์ต้องมี storage ภายนอกก่อนขึ้น Vercel
- driver pattern (`STORAGE_DRIVER`) จากบทที่แล้วออกดอก — สลับ local/supabase โดยไม่แตะโค้ดที่เรียกใช้

## 🔧 ปัญหาที่เจอจริงวันนี้ (troubleshooting)

1. **ต่อ Supabase ไม่ได้ ฟ้องรหัสผ่านผิด — ทั้งที่รหัสถูก**
   สาเหตุ: รหัสผ่าน DB มีตัว **`@`** อยู่ข้างใน. รูปแบบ connection string คือ `postgresql://user:password@host:port/db` — parser ใช้ `@` แบ่งรหัสผ่านกับ host, มี `@` ในรหัส = ตัด host ผิดตำแหน่ง → authentication failed / หา host ไม่เจอ
   วิธีแก้: **percent-encode อักขระพิเศษในรหัสผ่าน** — `@` → `%40` (ตัวอื่นก็เช่นกัน: `#` → `%23`, `:` → `%3A`; เช็คได้ด้วย `encodeURIComponent("รหัส")` ใน console). หรือตั้งรหัสใหม่ที่ไม่มีอักขระพิเศษตั้งแต่แรก
2. **Supabase project เดิม (ของพี่ต่าย) ใช้ไม่ได้** — ต่อไม่ติด กู้ไม่สำเร็จ → ตัดสินใจ**สร้าง project ใหม่**ใน Supabase แล้วเอา connection string ชุดใหม่มาใส่ `.env.supabase` แทน. บทเรียน: อย่าจมกับ infra เก่าที่กู้ไม่คุ้มเวลา — free tier สร้างใหม่เร็วกว่า (และจำไว้ว่า free tier จะ **pause DB ถ้าไม่มี traffic 7 วัน**)
