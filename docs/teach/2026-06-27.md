## บทที่ 1 — Register API: hash รหัสก่อนเก็บ แล้วออก session ให้เลย

### concept

สมัครสมาชิก = สร้าง user ใหม่ลง DB. กฎเหล็ก: **ห้ามเก็บรหัสผ่าน plaintext** ต้อง hash ก่อนเสมอ. สมัครเสร็จก็ตั้ง session cookie ให้เลย → ไม่ต้อง login ซ้ำ

### code — [app/api/auth/register/route.ts](../../app/api/auth/register/route.ts)

```ts
const { email, name, password } = result.data;   // ผ่าน registerSchema แล้ว

const existing = await prisma.user.findUnique({ where: { email } });
if (existing) return Response.json({ message: "อีเมลนี้ถูกใช้งานแล้ว" }, { status: 409 });

const passwordHash = await hashPassword(password);
const user = await prisma.user.create({
  data: { email, name, passwordHash },        // role default USER จาก schema
  select: { id: true, email: true, name: true, role: true },   // ไม่ select passwordHash
});

await setSessionCookie({ userId: user.id, email: user.email, role: user.role });
return Response.json(user, { status: 201 });
```

จุดน่าเรียน:

- **`409 Conflict`** = code มาตรฐานของ "ชนของเดิม" (email ซ้ำ); `201` = สร้างใหม่สำเร็จ
- **`select` ไม่เอา `passwordHash`** — Prisma default คืนทุก field รวม hash, ต้องตัดเองทุกครั้งที่ส่งกลับ client
- **role ไม่รับจาก body** — ปล่อย default `USER` ตาม schema. ถ้าเขียน `data: { ...body }` ผู้ใช้ส่ง `role: "ADMIN"` มาตั้งตัวเป็นแอดมินได้ (privilege escalation)
- **`registerSchema`** มี `confirmPassword` + `.refine(password === confirmPassword)` — เช็ครหัสตรงกันที่ schema เลย ([lib/schemas/auth.ts](../../lib/schemas/auth.ts))

### ⭐ takeaway

- ลำดับ register: validate → กันซ้ำ (`409`) → hash → create → set cookie → คืน user (ไม่มี hash)
- field อันตราย (`role`) ตั้งฝั่ง server เท่านั้น ห้ามรับจาก client
- ตรวจ 2 field พร้อมกัน (รหัส = ยืนยันรหัส) ใช้ `.refine()` ของ zod

## บทที่ 2 — Logout: ลบ cookie แล้วเด้งกลับ login

### concept

login = ตั้ง cookie, logout = **ลบ cookie ทิ้ง** เท่านั้นจริง ๆ

### code

**ฝั่ง API** — [app/api/auth/logout/route.ts](../../app/api/auth/logout/route.ts) (เมื่อวานเป็นฟังก์ชันว่าง):

```ts
export async function POST() {
  await clearSessionCookie();   // store.delete("session")
  return Response.json({ ok: true });
}
```

**ฝั่งปุ่ม** — [components/SiteHeader.tsx](../../components/SiteHeader.tsx) ผูก `onClick={handleLogout}`:

```tsx
const { mutateAsync } = useMutation({
  mutationFn: async () => {
    const res = await fetch("/api/auth/logout", { method: "POST" });
    if (!res.ok) throw new Error("Logout failed");
  },
});
const handleLogout = async () => {
  try { await mutateAsync(); router.push("/login"); }
  catch { toast.error("Logout failed. Please try again."); }
};
```

จุดน่าเรียน:

- แพตเทิร์นเดิมจากบท login: **ส่ง/แก้ = `useMutation`**, เช็ค `res.ok` เอง, try/catch แยกทางสำเร็จ/พลาด
- logout เป็น **POST ไม่ใช่ GET** — action ที่เปลี่ยน state ไม่ควรเป็น GET (prefetch/browser อาจยิงเองแล้วโดน logout)
- client ไม่ต้องแตะ cookie — server สั่งลบผ่าน response

### ⭐ takeaway

- logout = ลบ session cookie ฝั่ง server, จบ (ข้อดีของ httpOnly cookie: ไม่ต้องจัดการ token เองเลย)
- action ที่เปลี่ยน state ใช้ POST เสมอ

## บทที่ 3 — ฝั่งหน้าเว็บรู้ได้ไงว่าใคร login: `useMe`

### concept

session อยู่ใน httpOnly cookie ที่ **JS อ่านไม่ได้** (ตั้งใจกัน XSS). หน้าเว็บจะรู้ว่าใคร login / เป็น admin ไหม ต้องมี endpoint `/api/auth/me` ฝั่ง server อ่าน cookie ให้ แล้ว client ยิงถามผ่าน hook

### code — [lib/use-me.ts](../../lib/use-me.ts)

```ts
export function useMe() {
  const { data, isLoading } = useQuery<Me | null>({
    queryKey: ["me"],
    queryFn: async () => {
      const res = await fetch("/api/auth/me");
      if (res.status === 401) return null;    // ยังไม่ login = ไม่ใช่ error
      if (!res.ok) throw new Error("Failed to fetch user data");
      return res.json();
    },
  });
  return { user: data, isAdmin: data?.role === "ADMIN", isLoading };
}
```

**เอาไปใช้** — [app/menu/page.tsx](../../app/menu/page.tsx): ซ่อนปุ่ม Create Product ถ้าไม่ใช่ admin

```tsx
const { isAdmin } = useMe();
{isAdmin && <Link href="/menu/create" ...>Create Product</Link>}
```

จุดน่าเรียน:

- อ่านข้อมูล → **`useQuery`** (คู่กับ `useMutation` ที่ไว้เขียน)
- **`401` → `return null` ไม่ throw** — "ยังไม่ login" เป็นสถานะปกติ ไม่ใช่ error; throw ไว้ให้ error จริง (เน็ตล่ม)
- **`queryKey: ["me"]`** cache ไว้ที่เดียว → หลายหน้าเรียก `useMe()` ก็ยิง API จริงครั้งเดียว
- endpoint [app/api/auth/me/route.ts](../../app/api/auth/me/route.ts) `select` คืนแค่ `{ id, email, role }` — **ไม่เอา passwordHash**

### ⭐ takeaway

- cookie อ่านจาก JS ไม่ได้ → ต้องมี `/api/auth/me` ให้ client ถาม "ฉันเป็นใคร"
- อ่าน = `useQuery`; สถานะ "ยังไม่ login" (`401`) คืน `null`
- รวม logic ใน custom hook คืนค่าสำเร็จรูป (`isAdmin`)

## บทที่ 4 — กันประตูจริงอยู่ที่ server ไม่ใช่ที่ซ่อนปุ่ม: `requireRole`

### concept

บทที่ 3 ซ่อนปุ่ม Create Product จาก non-admin — แต่นั่นแค่ **UX**. คนที่ยิง API ตรง ๆ (devtools/Postman) ก็ POST `/api/product` สร้างสินค้าได้อยู่ดี. **ด่านจริงต้องอยู่ฝั่ง server** เช็ค session + role ก่อนทำงาน

### code

**helper** — [lib/auth/session.ts](../../lib/auth/session.ts):

```ts
export async function requireRole(role: SessionPayload["role"]) {
  const session = await getSession();
  if (!session) return Response.json({ message: "Unauthorized" }, { status: 401 });
  if (session.role !== role) return Response.json({ message: "Forbidden" }, { status: 403 });
  return session;
}
```

**กัน endpoint** — [app/api/product/route.ts](../../app/api/product/route.ts):

```ts
export async function POST(request: Request) {
  const auth = await requireRole("ADMIN");
  if (auth instanceof Response) return auth;   // ไม่ผ่าน → ส่ง 401/403 กลับเลย
  // ...ผ่านแล้วถึงสร้างสินค้า (GET ไม่กัน — ลูกค้าต้องดูเมนูได้)
}
```

จุดน่าเรียน:

- **`401` vs `403`**: `401 Unauthorized` = ไม่ได้ login (ไม่รู้เป็นใคร); `403 Forbidden` = login แล้วแต่สิทธิ์ไม่ถึง
- **`instanceof Response`** — `requireRole` คืน 2 แบบ: ผ่านคืน `session`, ไม่ผ่านคืน `Response` (ก้อน error). caller เช็คว่าได้ `Response` ไหม ถ้าใช่ return เลย ถ้าไม่ใช่ใช้ `session` ต่อ → guard reusable ทุก route
- ซ่อนปุ่ม (บท 3) + ล็อก API (บท 4) ทำ **คู่กัน**: ปุ่มเพื่อ UX, API เพื่อความปลอดภัยจริง

### ⭐ takeaway

- **ซ่อน UI ≠ ปลอดภัย** — endpoint ที่ต้องมีสิทธิ์ต้องเช็ค role ฝั่ง server เอง
- `401` = ไม่ได้ login, `403` = login แล้วสิทธิ์ไม่ถึง
- helper คืน `Response` ตอน fail → caller เช็ค `instanceof Response` แล้ว return ทันที

## บทที่ 5 — เริ่มงานอัปโหลดรูปสินค้า

### concept

สินค้าต้องมีรูป. ก้าวแรก: (1) เพิ่ม column เก็บ **ที่อยู่รูป** ใน DB, (2) helper รับไฟล์มาเซฟแล้วคืน URL, (3) endpoint รับไฟล์จากฟอร์ม

> เก็บแค่ **path/URL ของรูป** ลง DB ไม่ใช่ตัวไฟล์ — ไฟล์จริงไปอยู่ storage (ตอนนี้คือ `public/uploads/`)
> ⚠️ **สถานะวันนี้ = backend พร้อมแล้ว แต่ยังไม่ต่อ UI** — ยังไม่มีหน้า create/edit ไหนเรียก `/api/upload` หรือแสดง `imageUrl` (ยังไม่ wire)

### code

**1. field ใน schema** — [prisma/schema.prisma](../../prisma/schema.prisma) + migration `20260627074316_product_image`:

```prisma
model Product { /* ... */  imageUrl String?  /* nullable — สินค้าเก่าไม่มีรูปก็ผ่าน */ }
```
```sql
ALTER TABLE "products" ADD COLUMN "imageUrl" TEXT;
```

**2. helper เซฟไฟล์** — [lib/storage.ts](../../lib/storage.ts):

```ts
export async function saveImage(file: File) {
  if (!ALLOWED.includes(file.type)) throw new UploadError("รองรับเฉพาะ jpeg/png/webp/pdf");
  if (file.size > MAX_BYTES) throw new UploadError("ไฟล์ใหญ่เกิน 5MB");   // MAX = 5MB

  const ext = file.type.split("/")[1];
  const filename = `${crypto.randomUUID()}.${ext}`;    // ชื่อสุ่ม กันชน + กันเดา/path traversal
  const bytes = Buffer.from(await file.arrayBuffer());

  if (process.env.STORAGE_DRIVER === "supabase") {
    // upload to supabase  ← ยังเป็น stub
  } else {
    return saveToLocal(filename, bytes);    // เซฟลง public/uploads/, คืน /uploads/<ชื่อ>
  }
}
```

**3. endpoint** — [app/api/upload/route.ts](../../app/api/upload/route.ts):

```ts
const form = await request.formData().catch(() => null);
const file = form?.get("file");
if (!(file instanceof File)) return Response.json({ message: "ไม่พบไฟล์" }, { status: 400 });
const url = await saveImage(file);
return Response.json({ url });
```

จุดน่าเรียน:

- **`String?` (nullable)** = สินค้าเก่าไม่ต้องมีรูปก็ migrate ผ่าน (ถ้า required ต้อง backfill ของเก่าก่อน)
- **validate ก่อนเซฟเสมอ** — เช็คชนิด (allowlist) + ขนาด. ไฟล์จากผู้ใช้ = ขอบเขตที่ไว้ใจไม่ได้
- **ชื่อไฟล์สุ่ม `crypto.randomUUID()`** — ไม่ใช้ชื่อเดิมของผู้ใช้ (ชนกัน/มีอักขระอันตราย `../`)
- อัปโหลดไฟล์ใช้ **`request.formData()`** (multipart) ไม่ใช่ `request.json()` — JSON ส่ง binary ไม่ได้
- **`file instanceof File`** เช็คก่อนใช้ + TypeScript แคบ type ให้หลังบรรทัดนี้
- **`STORAGE_DRIVER`** เผื่อสลับไป supabase ทีหลังโดยไม่แก้ที่เรียกใช้

### ⭐ takeaway

- DB เก็บ path/URL ของรูป ไม่ใช่ตัวไฟล์; field รูปทำเป็น nullable
- ไฟล์จากผู้ใช้ต้อง validate ชนิด + ขนาด ก่อนเซฟ, ตั้งชื่อไฟล์ด้วยค่าสุ่ม
- อัปโหลด = `formData()` (multipart) ไม่ใช่ JSON

## 🔴 จุดอันตราย / งานที่ยังไม่จบ

โค้ดวันนี้เป็นสถานะกลางบทเรียน ยังมีช่องโหว่ที่รู้ตัว:

1. **login route ยังพัง (carryover จาก [2026-06-12](./2026-06-12.md) — วันนี้ยังไม่แตะ)** — ใน [app/api/auth/login/route.ts](../../app/api/auth/login/route.ts) บรรทัด `verifyPassword` ยัง comment อยู่ + เช็คแค่ `!user` → **รู้ email ก็ login ได้ ไม่ต้องรู้รหัส**. และยังมี `console.log("Received login data", { email, password })` = รหัส plaintext โผล่ใน log. ต้องเปิด verify + ลบ log:
   ```ts
   if (!user || !(await verifyPassword(password, user.passwordHash))) {
     return Response.json({ message: "Invalid email or password" }, { status: 401 });
   }
   ```
2. **`JWT_SECRET || "default_secret"`** ใน [lib/auth/session.ts](../../lib/auth/session.ts) — ย้าย secret มา `.env` แล้ว (ดีขึ้นจากเมื่อวานที่ hardcode) **แต่ยังมี fallback ค่าตายตัวที่ใครก็เดาได้**. ถ้า prod ลืมตั้ง `JWT_SECRET` ระบบเงียบ ๆ ใช้ `"default_secret"` → ใครก็ปลอม token ตั้งตัวเป็น admin ได้. ควร throw ตอน start ถ้าไม่มี secret (เวอร์ชัน `prep/auth-rebased` ใช้ `SESSION_SECRET` แบบไม่มี fallback)
3. **`/api/upload` ไม่เช็คสิทธิ์เลย** — ใครก็ได้ (ไม่ต้อง login) อัปไฟล์ 5MB เข้า server ไม่จำกัด = พื้นที่เต็ม/ค่าใช้จ่ายบาน. ต้องใส่ `requireRole("ADMIN")` เหมือน `/api/product` (บท 4)
4. **`"application/octet-stream"` อยู่ใน `ALLOWED`** ของ [lib/storage.ts](../../lib/storage.ts) — เป็นชนิด "ไฟล์อะไรก็ไม่รู้" ที่ browser ส่งมาเวลาเดาชนิดไม่ออก → ใส่ในรายการอนุญาต = เปิดช่องอัปไฟล์อะไรก็ได้ทะลุ allowlist. ควรตัดออก
5. **สาขา supabase ใน `saveImage` ยังว่าง** — ถ้าตั้ง `STORAGE_DRIVER=supabase` ฟังก์ชันจะ return `undefined` (ไม่เซฟ ไม่ error). อย่าเพิ่งเปิดใช้
