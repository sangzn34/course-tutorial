## บทที่ 1 — Server Component กับ Client Component (concept สำคัญที่สุดวันนี้)

### concept

ใน Next.js App Router ทุกไฟล์ใน `app/` เป็น **Server Component โดยปริยาย** — โค้ดรันบน server, ส่ง HTML สำเร็จรูปมาให้ browser. ข้อดี: เร็ว, ดึงข้อมูล/ความลับ (เช่น DB) ได้ตรง ๆ.

แต่ Server Component **ใช้ของที่ต้องรันใน browser ไม่ได้** — เช่น `useState`, `onClick`, `useEffect`. ถ้าต้องการพวกนี้ ต้องเติม `"use client"` บรรทัดแรกของไฟล์ → กลายเป็น **Client Component**.

> หลักคิด: ทำให้ component เป็น **server ให้มากที่สุด** แล้ว "ดัน" เฉพาะส่วนที่ต้อง interactive ลงไปเป็น client ที่ leaf (ใบล่างสุด ของต้นไม้ component)

### สิ่งที่ทำในโปรเจค

เดิม [app/layout.tsx](../../app/layout.tsx) มี `"use client"` ทั้งไฟล์ — ทำให้ทั้งหน้าเป็น client โดยไม่จำเป็น. วันนี้เอา `"use client"` ออก แล้วดัน logic ที่ต้อง interactive ไปไว้ component ลูกแทน

**ทำไมต้องเอาออก** — เพราะ Next.js มีของชื่อ `metadata` (ตั้ง title/description ของหน้าให้ SEO) ที่ **export ได้จาก Server Component เท่านั้น**:

```ts
// app/layout.tsx — ใช้ได้เพราะตอนนี้เป็น Server Component แล้ว
export const metadata: Metadata = {
  title: { default: "My Coffee", template: "%s - My Coffee" },
  description: "ร้านกาแฟออนไลน์ - สั่งง่าย ส่งไว",
};
```
(`template: "%s - My Coffee"` = หน้าอื่นตั้ง title ว่า "เมนู" จะกลายเป็น "เมนู - My Coffee" อัตโนมัติ)

ส่วนที่ต้อง interactive ก็ย้ายไปไฟล์ลูกที่มี `"use client"` เอง เช่น [lib/client.tsx](../../lib/client.tsx) (ตัวห่อ React Query — ต้องเป็น client เพราะใช้ `useState`)

### ⭐ takeaway

- ไฟล์ใน `app/` = Server Component จนกว่าจะเติม `"use client"`
- เติม `"use client"` **เฉพาะไฟล์ที่ต้องใช้ state/event/effect จริง ๆ** อย่าเหวี่ยงใส่ทั้ง layout
- เห็น error "you're importing a component that needs useState... mark it use client" = เจอเส้นแบ่งนี้

## บทที่ 2 — เปลี่ยน Sidebar เป็น SiteHeader

### concept

"shell" = โครงที่อยู่ทุกหน้า (header/sidebar/footer). โปรเจคนี้เคยใช้ sidebar ด้านซ้าย. ร้านค้าออนไลน์คนนิยมใช้ **top bar** มากกว่า (ลูกค้าคุ้น, เหลือพื้นที่เนื้อหากว้าง) เลยเปลี่ยน

### สิ่งที่ทำ

สร้าง [components/SiteHeader.tsx](../../components/SiteHeader.tsx) — แถบบนสุดที่มี: โลโก้ → เมนู → (ดันขวา) → ปุ่มธีม + ปุ่มตะกร้า

> ไฟล์นี้มี `"use client"` เพราะใช้ `usePathname()` (hook) — ตรงกับบทที่ 1: component ที่ต้องใช้ hook = client

**โครงด้วย flexbox:**
```tsx
<header className="sticky top-0 z-40 border-b bg-background/80 backdrop-blur ...">
  <div className="mx-auto flex h-14 max-w-6xl items-center gap-3 px-4 sm:px-6">
    <Link href="/">...โลโก้...</Link>
    <nav className="ml-2 flex ...">...เมนู...</nav>
    <div className="ml-auto flex items-center gap-1.5">  {/* ml-auto = ดันไปขวาสุด */}
      <ThemeToggle />
      <CartButton />
    </div>
  </div>
</header>
```
- `flex items-center` = เรียงของแนวนอน จัดกึ่งกลางแนวตั้ง
- `h-14` = สูงคงที่ 56px
- **`ml-auto`** = เคล็ดลับ: `margin-left: auto` ดันชิ้นนี้ (และที่เหลือ) ไปขวาสุด → โลโก้/เมนูอยู่ซ้าย, ธีม/ตะกร้าอยู่ขวา

**1. แถบติดบนเวลาเลื่อน + เบลอพื้นหลัง**
```
sticky top-0 z-40 bg-background/80 backdrop-blur supports-backdrop-filter:bg-background/60
```
- `sticky top-0` = ค้างบนสุดเวลา scroll (ต่างจาก `fixed` ตรงที่ยังกินพื้นที่ใน layout ปกติ)
- `z-40` = ลอยทับเนื้อหาที่เลื่อนผ่านใต้มัน
- `bg-background/80` = สีพื้น (token) opacity 80% → โปร่งนิด ให้เห็นของที่เบลออยู่ข้างหลัง
- `backdrop-blur` = เบลอสิ่งที่อยู่ใต้แถบ (เอฟเฟกต์กระจกฝ้า)
- `supports-backdrop-filter:...` = ถ้า browser รองรับ backdrop-filter ค่อยใช้สูตรโปร่งกว่า (`/60`) — เป็น progressive enhancement

**2. รู้ว่าตอนนี้อยู่หน้าไหน** ด้วย hook `usePathname()` (คืน path ปัจจุบัน เช่น `/menu`)
```tsx
const pathname = usePathname();
// ...ใน nav loop:
const active = pathname === item.href || pathname.startsWith(`${item.href}/`);
```
→ เมนูของหน้าปัจจุบันจะ highlight. เช็ค 2 แบบ:
- `=== item.href` = ตรงเป๊ะ (อยู่ `/menu`)
- `.startsWith(item.href + "/")` = อยู่หน้าลูก (เช่น `/menu/123` ก็ยังให้เมนู "เมนู" highlight)

แล้วใช้ `cn()` สลับ class ตาม `active`:
```tsx
className={cn(
  "rounded-md px-2.5 py-1.5 text-sm",
  active ? "bg-muted text-foreground font-medium"
         : "text-muted-foreground hover:text-foreground hover:bg-muted/60",
)}
```
> `cn()` (จาก [lib/utils.ts](../../lib/utils.ts)) = ฟังก์ชันรวม class แบบฉลาด — ต่อ string ได้ + จัดการ class ที่ชนกันให้

**3. ซ่อน header บนหน้า login/register**
```tsx
if (pathname === "/login" || pathname === "/register") return null;
```
→ หน้า login/register จะดูสะอาด เป็น "หน้าจอเดี่ยว" ไม่มี nav มากวน. **component คืน `null` = ไม่ render อะไรเลย** (header หายทั้งแถบ)

### ⭐ takeaway

- **`flex` + `ml-auto`** = แพตเทิร์นมาตรฐานของ header: ของกลุ่มซ้าย / ดันกลุ่มขวาด้วย `ml-auto`
- **`sticky top-0 z-40`** = แถบติดบนเวลาเลื่อน (ใช้กับ header/toolbar)
- `usePathname()` (Client Component เท่านั้น) → รู้ URL ปัจจุบัน → ทำ active state / ซ่อนแสดงตาม path
- `return null` = บอก React ว่า "ไม่ต้องวาดอะไร"
- ใช้ `cn(base, cond ? a : b)` สลับ class ตาม state — อย่าต่อ string เอง

## บทที่ 3 — Dark Mode ด้วย next-themes (+ บั๊กที่เจอจริงวันนี้)

### concept

dark mode ใน Tailwind/shadcn ทำงานด้วยการเติม class `dark` ที่ tag `<html>`. พอมี `.dark` สีทั้งหน้าจะสลับ (เพราะสีถูกผูกไว้กับ "token" ที่เปลี่ยนค่าตาม `.dark`). เราใช้ library `next-themes` มาจัดการการเติม/ถอด class นี้ให้

ติดตั้ง: `next-themes@^0.4.6`

### โครงสร้าง 3 ชิ้น

**1. ตัวห่อ provider** — [components/ThemeProvider.tsx](../../components/ThemeProvider.tsx)
```tsx
<NextThemes attribute="class" defaultTheme="system" enableSystem disableTransitionOnChange>
```
- `attribute="class"` = ให้ next-themes สลับธีมด้วย **class** `dark` (ตรงกับวิธีที่ shadcn ออกแบบสีไว้)
- `defaultTheme="system" enableSystem` = ตามค่าเครื่องผู้ใช้ (ตั้ง dark ไว้ก็ dark)
- `disableTransitionOnChange` = ปิด animation ตอนสลับ กันสีค่อย ๆ ไล่ (กระพริบน่าเกลียด)

**2. ปุ่มสลับ** — [components/ThemeToggle.tsx](../../components/ThemeToggle.tsx)
- ใช้ `resolvedTheme` ไม่ใช่ `theme` — เพราะ `theme` อาจเป็นคำว่า `"system"` (ไม่รู้ว่ามืดหรือสว่าง), ส่วน `resolvedTheme` คือผลจริง `"light"`/`"dark"`
- วาด **ทั้งไอคอน Sun และ Moon พร้อมกัน** แล้วซ่อน/โชว์ด้วย CSS (`scale`, `rotate`, `opacity`)

> **ทำไมไม่ใช้ `{isDark ? <Moon/> : <Sun/>}`?** — ตอน server render ยังไม่รู้ธีมของ browser → ถ้าเลือกวาดอันเดียวตามธีม จะได้ HTML จาก server ไม่ตรงกับที่ browser อยากวาด เกิดอาการ "hydration mismatch" (React เตือน/วาดเพี้ยน). วาดทั้งคู่ไว้แล้วใช้ CSS ซ่อน = HTML เหมือนกันทุกฝั่ง ปลอดภัยกว่า

**3. ใส่ provider ใน layout** — และเติม `suppressHydrationWarning` ที่ `<html>`
```tsx
<html lang="en" suppressHydrationWarning>
```
> next-themes จงใจแก้ class ที่ `<html>` ก่อน React จะ hydrate → markup server/client ต่างกัน **โดยตั้งใจ** → `suppressHydrationWarning` บอก React ว่า "อันนี้รู้แล้ว ไม่ต้องเตือน"

### 🐛 บั๊กที่เจอจริงวันนี้ — กดปุ่มแล้วธีมไม่เปลี่ยน

**อาการ**: กดปุ่มสลับธีมแล้วไม่มีอะไรเกิดขึ้น — class `dark` ไม่ถูกใส่ที่ `<html>`

**สาเหตุ**: ใน layout เผลอ import `ThemeProvider` ผิดตัว
```tsx
// ❌ ผิด — ตัว provider ดิบจาก library ยังไม่มี config (attribute="class" ฯลฯ)
import { ThemeProvider } from "next-themes";

// ✅ ถูก — ตัวห่อของเราที่ตั้ง attribute="class" ไว้แล้ว
import { ThemeProvider } from "@/components/ThemeProvider";
```

**ทำไมถึงพลาดง่าย**: ชื่อ export ซ้ำกันเป๊ะ (`ThemeProvider` ทั้งคู่) → autocomplete หยิบผิดได้ง่าย และ **TypeScript ไม่ฟ้อง** เพราะทั้งสองตัวก็เป็น provider ที่ใช้งานได้ — แค่ตัวดิบไม่มี config เลยไม่สลับ class

**บทเรียน**:
- import ผิดที่ บางทีไม่ error แต่ "เงียบ ๆ ไม่ทำงาน" — ดูยากกว่า error เสียอีก
- เวลามี wrapper component ของเราเอง ให้ import จากตัว wrapper เสมอ (`@/components/...`); จาก library ดิบเอาเฉพาะ **hook** (`useTheme`) ก็พอ

## บทที่ 4 — หน้าแรกเป็น Landing Page + ดึงข้อมูลด้วย React Query

### concept

หน้าแรก ([app/page.tsx](../../app/page.tsx)) เปลี่ยนจากฟอร์ม login เป็นหน้าโปรโมตร้าน เรียงเป็น **3 section จากบนลงล่าง**:

```
┌─────────────────────────────────────┐
│  Section 1: Hero (พาดหัว + ปุ่ม + รูป) │
├─────────────────────────────────────┤
│  Section 2: Features (จุดเด่น 3 ข้อ)  │
├─────────────────────────────────────┤
│  Section 3: เมนูแนะนำ (ดึงจาก API)    │
└─────────────────────────────────────┘
```

โครงรวมเป็น `<div className="flex flex-1 flex-col">` (เรียง section แนวตั้ง) แล้วแต่ละ section เป็น `<section>`

**pattern ที่ทุก section ใช้เหมือนกัน — กล่องจำกัดความกว้างตรงกลาง:**
```tsx
<div className="mx-auto max-w-6xl px-4 sm:px-6">
```
- `max-w-6xl` = กว้างสุดไม่เกิน ~72rem (จอใหญ่เนื้อหาไม่ยืดเต็มจนอ่านยาก)
- `mx-auto` = จัดกล่องไว้กลางจอ
- `px-4 sm:px-6` = เว้นขอบซ้ายขวา 16px บนมือถือ, 24px บนจอ ≥ small (กันเนื้อหาชิดขอบ)

> 💡 เรื่อง responsive ของ Tailwind: class เปล่า ๆ (`px-4`) = ใช้ทุกขนาดจอ; เติม prefix (`sm:` `lg:`) = ใช้**ตั้งแต่จอขนาดนั้นขึ้นไป**. เป็นแนวคิด **mobile-first** — เขียนมือถือก่อน แล้วค่อย override จอใหญ่

### Section 1 — Hero (บรรทัด 98–134)

ส่วนพาดหัวบนสุด: ข้อความ+ปุ่มด้านซ้าย, รูปกาแฟด้านขวา

**สลับ layout ตามขนาดจอด้วย flex:**
```tsx
<div className="... flex flex-col gap-6 lg:flex-row lg:items-center">
```
- `flex flex-col` = มือถือ: เรียง**บนลงล่าง** (ข้อความอยู่บน รูปอยู่ล่าง)
- `lg:flex-row` = จอใหญ่: เรียง**ซ้ายไปขวา** (ข้อความซ้าย รูปขวา)
- `gap-6` = ระยะห่างระหว่างชิ้น

**ฝั่งซ้าย** — badge + headline + คำโปรย + ปุ่ม:
```tsx
<Badge variant="secondary">เปิดบริการทุกวัน 7:00–20:00</Badge>
<h1 className="text-4xl font-bold sm:text-5xl">
  กาแฟดี ๆ<br/>
  <span className="text-primary">ส่งถึงโต๊ะคุณ</span>   {/* สีเน้นจาก token */}
</h1>
<p className="text-muted-foreground max-w-md">เมล็ดคั่วเองทุกล็อต...</p>
```
- `text-4xl ... sm:text-5xl` = ตัวอักษรใหญ่ขึ้นบนจอใหญ่
- `text-primary` / `text-muted-foreground` = สีจาก token (เน้น / สีจาง) ไม่ hardcode
- ปุ่ม 2 อันใช้ `Button` + `render={<Link/>}` (pattern เดียวกับบทที่ 1 — ปุ่มที่ทำงานเป็นลิงก์)

**ฝั่งขวา** — กล่องไล่สีพร้อมไอคอนกาแฟใหญ่:
```tsx
<div className="bg-linear-to-br from-primary/20 via-primary/5 to-transparent rounded-2xl ...">
  <Coffee className="text-primary/60 h-32 w-32" />
</div>
```
- `bg-linear-to-br from-... via-... to-transparent` = ไล่สีทแยงมุม (เป็น placeholder แทนรูปจริง)
- `/20` `/60` ต่อท้ายสี = ความโปร่งใส (opacity 20% / 60%)

### Section 2 — Features (จุดเด่น 3 ข้อ, บรรทัด 135–149)

แถบบอกจุดขายร้าน 3 ข้อ. ข้อมูลแยกเป็น **array** ไว้บนไฟล์ แล้ว `.map()` วาด → เพิ่ม/แก้จุดเด่นแค่แก้ array ไม่ต้องแตะ JSX:
```tsx
const features = [
  { icon: Coffee, title: "เมนูสด",   body: "คั่วใหม่ทุกวัน..." },
  { icon: Truck,  title: "จัดส่งเร็ว", body: "30 นาทีถึงมือ" },
  { icon: Wallet, title: "ชำระสะดวก", body: "PromptPay / บัตร..." },
] as const;
```

**วางเป็น grid responsive:**
```tsx
<div className="grid grid-cols-1 gap-6 sm:grid-cols-3">
  {features.map(({ icon: Icon, title, body }) => (
    <div key={title} className="flex gap-3"> ... </div>
  ))}
</div>
```
- `grid-cols-1` = มือถือ: 1 คอลัมน์ (เรียงลงมา)
- `sm:grid-cols-3` = จอ ≥ small: 3 คอลัมน์ (เรียงข้าง)
- `{ icon: Icon }` = rename ตอน destructure เป็นตัวพิมพ์ใหญ่ `Icon` — เพราะ React component ต้องขึ้นต้นตัวใหญ่ถึงจะ `<Icon/>` ได้
- `key={title}` = ทุกอย่างที่ `.map()` ต้องมี `key` (ให้ React track แต่ละชิ้น)

### Section 3 — เมนูแนะนำ (ดึงจาก API, บรรทัด 150–175)

ส่วนนี้ต้องโหลดสินค้าจริงจาก API มาโชว์ — ใช้ **React Query** (`useQuery`) จัดการ loading/cache/error ให้:
```tsx
const { data: products = [] } = useQuery<Product[]>({
  queryKey: ["product"],                       // "ชื่อ" ของข้อมูลก้อนนี้ใน cache
  queryFn: () =>
    fetch(`/api/product`).then((res) => {
      if (!res.ok) throw new Error("โหลดสินค้าไม่สำเร็จ");
      return res.json();
    }),
});
```
- `data: products = []` = ตั้ง default array ว่าง กัน `undefined` ตอนยังโหลดไม่เสร็จ
- `queryKey: ["product"]` = ชื่อ cache; หน้าอื่นเรียก `useQuery` ด้วย key เดียวกัน = ใช้ cache ร่วม ไม่ยิงซ้ำ

**โชว์เฉพาะเมื่อมีของ — conditional render:**
```tsx
{products?.length > 0 && (
  <section> ... </section>
)}
```
> `{เงื่อนไข && <JSX/>}` = ถ้าเงื่อนไขจริง วาด JSX, ถ้าเท็จไม่วาดอะไร. ตรงนี้ = ไม่มีสินค้าก็ไม่โชว์ section นี้เลย

**วาดการ์ดแบบ grid 1→2→3 คอลัมน์:**
```tsx
<div className="grid grid-cols-1 gap-4 sm:grid-cols-2 lg:grid-cols-3">
  {products?.map((p) => (
    <ProductCard key={p.id} id={p.id}
      name={p.name ?? ""} price={p.price} description={p.description ?? ""} />
  ))}
</div>
```
- `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3` = มือถือ 1, แท็บเล็ต 2, จอใหญ่ 3 คอลัมน์
- `p.name ?? ""` = ถ้า `name` เป็น null/undefined ใช้ string ว่างแทน (`??` = nullish coalescing)

### ⚠️ จุดที่ต้องตามแก้ (เจอตอนสอน ยังค้างอยู่)

1. มี `console.log("products", products)` ค้างใน [app/page.tsx](../../app/page.tsx) — ลบทิ้ง (ตอน `pnpm build` มันโผล่ออกมาด้วย)
2. โค้ดฟอร์ม login เดิมถูก **comment ทิ้งไว้** ท้ายไฟล์ (บรรทัด 179–205) — ควรลบ หรือย้ายไปหน้า `/login` จริง
3. ใน Tailwind v4 ชื่อ class gradient คือ `bg-linear-to-br` (เวอร์ชัน 3 ใช้ `bg-gradient-to-br`) — ระวังถ้าก๊อปโค้ดเก่ามาวาง

### ⭐ takeaway

- **แยกหน้าเป็น section** — แต่ละ `<section>` ทำหน้าที่เดียว, ใช้กล่อง `mx-auto max-w-6xl px-4` ครอบเหมือนกันทุกอัน
- **responsive = mobile-first**: เขียน layout มือถือก่อน (`flex-col`, `grid-cols-1`) แล้ว override จอใหญ่ด้วย prefix (`lg:flex-row`, `sm:grid-cols-3`)
- **ข้อมูลที่ซ้ำ ๆ ทำเป็น array แล้ว `.map()`** (features, products) — อย่าก๊อป JSX ซ้ำ; อย่าลืม `key`
- **ข้อมูลจาก API → `useQuery`** แทน `fetch`+`useState`+`useEffect` เอง (ได้ cache + loading/error ฟรี); ตั้ง default (`= []`) เสมอ
- **conditional render** `{cond && <JSX/>}` = ซ่อน/โชว์ส่วนของหน้าตามเงื่อนไข

## บทที่ 5 — ขัดปุ่ม Cart ให้ใช้ component กลาง (shadcn/ui)

### concept

เดิมปุ่ม/drawer ของตะกร้าเขียน `<button>` กับใส่สี hardcode เอง (`bg-blue-600`). เปลี่ยนมาใช้ component กลางจาก shadcn/ui → สีตาม token (สลับ dark mode ได้เอง), หน้าตาสม่ำเสมอทั้งเว็บ

ตะกร้าทำงานร่วมกัน 3 ไฟล์:
```
lib/cart.ts          → state กลาง (Jotai atom) — ของในตะกร้า + ตัวนับ
  ├─ CartButton.tsx  → ปุ่มใน" header (โชว์จำนวน, เปิด drawer)
  └─ CartDrawer.tsx  → แผงเลื่อนจากขวา (แก้จำนวน, ลบ, ไป checkout)
```

---

### [components/CartButton.tsx](../../components/CartButton.tsx) — ปุ่มตะกร้า

```tsx
export const CartButton = () => {
  const count = useAtomValue(cartCountAtom);     // อ่านตัวนับจาก state กลาง
  const [open, setOpen] = useState(false);        // คุมว่า drawer เปิดอยู่ไหม

  return (
    <>
      <Button variant="ghost" size="icon-sm"
        aria-label={`ตะกร้า (${count} รายการ)`}
        onClick={() => setOpen(true)} className="relative">
        <ShoppingCart className="h-4 w-4" />
        {count > 0 && (
          <span className="bg-destructive ... absolute -top-1 -right-1 ... tabular-nums">
            {count > 99 ? "99+" : count}
          </span>
        )}
      </Button>
      <CartDrawer open={open} onOpenChange={setOpen} />
    </>
  );
};
```

จุดน่าเรียน:
- **`useAtomValue(cartCountAtom)`** — อ่านตัวนับจาก Jotai (อ่านอย่างเดียว). `cartCountAtom` เป็น *derived* — รวม `quantity` ทุกชิ้นใน `cartAtom` ให้เอง พอเพิ่มของในตะกร้า ตัวเลขบนปุ่มขยับเองทันที โดยปุ่มไม่ต้องรู้ logic
- **ปุ่มถือ state `open` ของ drawer เอง** (`useState`) แล้วส่งลงไปให้ drawer
- **badge นับจำนวน** — โชว์เฉพาะ `count > 0` (conditional render):
  - `absolute -top-1 -right-1` = วาง badge มุมขวาบนของปุ่ม (ปุ่มต้องมี `relative` ครอบ — `absolute` อิงกับ `relative` ที่ใกล้สุด)
  - `bg-destructive` = สีแดงเตือน (token ไม่ใช่ `bg-red-500`)
  - `tabular-nums` = ตัวเลขทุกตัวกว้างเท่ากัน → badge ไม่ขยับตอนเลขเปลี่ยน
  - `count > 99 ? "99+" : count` = กันเลขล้นกล่อง

---

### [components/CartDrawer.tsx](../../components/CartDrawer.tsx) — แผงตะกร้า

**1. เป็น "controlled component"** — drawer ไม่ถือ state เปิด/ปิดเอง แต่รับมาจาก parent:
```tsx
type CartDrawerProps = {
  open: boolean;                          // parent บอกว่าเปิดอยู่ไหม
  onOpenChange: (open: boolean) => void;  // drawer บอก parent เมื่ออยากเปลี่ยน
};
```
> นี่คือแพตเทิร์น **controlled component** — "ของจริง" (state) อยู่ที่ parent, ลูกแค่รับค่ามาแสดง + ส่ง event กลับ. ชื่อ prop `open` / `onOpenChange` เป็นมาตรฐานของ shadcn `Sheet` พอดี เลยส่งต่อตรง ๆ ได้

**2. ใช้ `Sheet` ของ shadcn** (แผงเลื่อนจากขอบจอ) แทน markup เลื่อนเอง:
```tsx
<Sheet open={open} onOpenChange={onOpenChange}>
  <SheetContent side="right" ...>      {/* เลื่อนเข้าจากขวา */}
    <SheetHeader>...</SheetHeader>
    ...รายการ...
    <SheetFooter>...ยอดรวม + ปุ่ม checkout...</SheetFooter>
  </SheetContent>
</Sheet>
```
> `Sheet` จัดการ animation เลื่อนเข้า/ออก + ฉากหลัง + ปิดด้วย Esc/คลิกนอก ให้หมด — ไม่ต้องเขียน `translate-x` เอง (เทียบกับเวอร์ชันเก่าในบันทึก [2026-05-23](./2026-05-23.md) ที่เขียน drawer มือ)

**3. อ่าน+เขียน state ตะกร้า** ด้วย `useAtom` (ได้ทั้งค่าและฟังก์ชันเซ็ต):
```tsx
const [carts, setCarts] = useAtom(cartAtom);
const total = carts.reduce((s, c) => s + c.price * c.quantity, 0);   // ยอดรวม
```

**4. ลบ / แก้จำนวน — แก้แบบ immutable (สร้าง array ใหม่เสมอ):**
```tsx
// ลบ: เก็บเฉพาะตัวที่ไม่ใช่ id นี้
const remove = (id) => setCarts(carts.filter((c) => c.productId !== id));

// +/-: หาตัวที่ตรง id แล้วบวก delta, จบด้วยกรองตัวที่เหลือ 0 ออก
const setQty = (id, delta) =>
  setCarts(
    carts
      .map((c) => c.productId === id ? { ...c, quantity: c.quantity + delta } : c)
      .filter((c) => c.quantity > 0),   // กดลบจนเหลือ 0 = หายจากตะกร้าเอง
  );
```

**5. ปุ่มไป checkout เป็นลิงก์** (pattern `render` เดียวกับบทที่ 1):
```tsx
<Button nativeButton={false} render={<Link href="/checkout" />}>
  ดำเนินการชำระเงิน
</Button>
```

### ⭐ takeaway

- **derived state** (`cartCountAtom`) — คำนวณจาก state อื่น ไม่เก็บซ้ำ; แหล่งความจริงมีที่เดียว (`cartAtom`)
- **controlled component** — ของที่เปิด/ปิด (drawer, modal) มักให้ parent ถือ state แล้วส่ง `open` / `onOpenChange` ลงมา
- ใช้ component สำเร็จ (`Sheet`) แทนเขียน animation/overlay เอง — ได้ a11y (Esc, focus trap) ฟรี
- แก้ array ใน state = `map`/`filter`/spread เสมอ (immutable) ไม่แก้ของเดิม
- `absolute` วางตำแหน่งอิง `relative` ตัวที่ใกล้สุด — ใช้ทำ badge มุมปุ่ม

### ⚠️ ของเฉพาะโปรเจคนี้ — ใช้ `render` ไม่ใช่ `asChild`

shadcn/ui ปกติทำปุ่มให้เป็นลิงก์ด้วย prop `asChild`. **แต่เวอร์ชันที่ลงในโปรเจคนี้ (v4.7.0) สร้างบน Base UI** ซึ่งใช้ prop ชื่อ `render` แทน:
```tsx
<Button nativeButton={false} render={<Link href="/checkout" />}>
  ดำเนินการชำระเงิน
</Button>
```
- `render={<Link .../>}` = "ปุ่มนี้จริง ๆ ให้ render เป็น `<Link>`" (ได้สไตล์ปุ่ม แต่ทำงานเป็นลิงก์)
- `nativeButton={false}` = บอกว่า element จริงไม่ใช่ `<button>`

> ถ้าก๊อปตัวอย่าง shadcn จากเน็ตที่ใช้ `asChild` มาวาง → จะไม่ทำงาน เพราะโปรเจคนี้ใช้ `render`

[components/ProductCard.tsx](../../components/ProductCard.tsx) — แก้ `price.toFixed()` เป็น `price?.toFixed()` (`?.` กัน error ตอน `price` ยังไม่มา = undefined)

## บทที่ 6 — Locator loader ให้รันเฉพาะตอน dev

### concept

โปรเจคนี้ใช้ LocatorJS (เครื่องมือ dev: คลิก element ในเว็บแล้วกระโดดไปไฟล์ source ใน editor). มันทำงานด้วย "loader" ที่เติม attribute พิเศษ (`data-locatorjs`) ลงทุก element เวลา build

attribute พวกนี้มีไว้ debug ตอนพัฒนา — **ไม่ควรติดไปกับเว็บจริง (production)**. เลยตั้งให้ loader ทำงานเฉพาะตอน `next dev`

### โค้ด — [next.config.ts](../../next.config.ts)

Next.js 16 (bundler ชื่อ Turbopack) มี "condition" ในตัวสำหรับจำกัดว่า loader รันเมื่อไหร่:
```ts
turbopack: {
  rules: {
    "**/*.{tsx,jsx}": {
      condition: { all: ["development", { not: "foreign" }] },
      loaders: [{ loader: "@locator/webpack-loader", options: { env: "development" } }],
    },
  },
}
```

| condition | แปลว่า |
|---|---|
| `"development"` | รันเฉพาะตอน `next dev`; ตอน `next build` (production) จะข้าม |
| `{ not: "foreign" }` | ไม่ต้องแตะไฟล์ใน `node_modules` (โค้ด library) → build เร็วขึ้น |

### ⭐ takeaway + ข้อควรระวังเรื่องความซื่อสัตย์

- Turbopack รู้ context ของตัวเอง (`development` / `production` / `browser` / `node`) → ประกาศ `condition` ตรง ๆ สะอาดกว่าการไปเช็ค `process.env.NODE_ENV` เอง
- **การ gate นี้เป็น optimization ไม่ใช่ bugfix** — ทดสอบแล้ว (`NODE_ENV=production pnpm build`) ว่า ถึงไม่ gate, prod build ก็ยังผ่านบน Next 16.2.6 ไม่ได้พังจาก loader ตัวนี้

> 📝 บทเรียนสำหรับคนเรียน: ตอนแรกบันทึกนี้เคยเขียนว่า "loader ทำให้ prod build พังด้วย error `Moon is not defined`" — แต่พอ**ลองรันจริง** กลับ build ผ่าน และอ่าน source ของ loader ก็พบว่าไม่ได้ทำงานอย่างที่เดา. เลยแก้บันทึกให้ตรงความจริง. **อย่าเชื่อคำอธิบายสาเหตุที่ยังไม่ได้พิสูจน์ — ลองรัน/อ่าน code ก่อนเสมอ**

## บทที่ 7 — บังคับใช้ pnpm ตัวเดียว (กันปัญหา "เครื่องฉันรันได้ เครื่องเธอพัง")

### concept

โปรเจค Node มี "package manager" หลายตัว (npm / yarn / pnpm). แต่ละตัวสร้างไฟล์ lock ของตัวเอง (`package-lock.json` / `yarn.lock` / `pnpm-lock.yaml`) ที่ตรึงเวอร์ชัน dependency เป๊ะ ๆ. **ถ้าใช้ปนกัน เวอร์ชันจะเพี้ยนกันคนละเครื่อง** → บั๊กแบบ "เครื่องฉันรันได้แต่เครื่องเพื่อนพัง"

### สิ่งที่ทำ

- ลบ `package-lock.json` (ของ npm, 14530 บรรทัด) ทิ้ง — โปรเจคนี้ใช้ **pnpm เท่านั้น**
- [.gitignore](../../.gitignore): กัน `package-lock.json` + `yarn.lock` ไม่ให้ถูก commit อีก
- [package.json](../../package.json): ระบุ `"packageManager": "pnpm@10.33.0"` ให้ชัด
- ตรึงเวอร์ชัน Tailwind เป๊ะ: `^4` → `4.3.0`

### ⭐ takeaway

- หนึ่งโปรเจค = หนึ่ง package manager = หนึ่ง lockfile. เลือกตัวไหนแล้วใช้ตัวนั้น และ commit lockfile ของมันเข้า git ด้วย
- ติดตั้ง dependency ใช้คำสั่งของ manager นั้น (`pnpm install` ไม่ใช่ `npm install`)
