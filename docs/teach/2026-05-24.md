### ⚠️ จุดอันตราย ต้องตามแก้

1. **Plaintext password** — login query `where: { email, passwordHash: password }` ตรงๆ ต้อง bcrypt + เทียบ hash ฝั่ง server
2. **Credential leak** — `console.log("Received login data:", { username, password })` → ลบทิ้ง
3. **No session** — `logout` / `me` ยังว่าง ต้องเลือก strategy (JWT cookie / iron-session / NextAuth) ก่อน

### Product: soft delete + partial unique index

```prisma
model Product {
  name      String
  deletedAt DateTime?

  @@unique([name], where: { deletedAt: null })  // unique เฉพาะแถวที่ยังไม่ลบ
}
```

เปิด preview feature:

```prisma
generator client {
  previewFeatures = ["partialIndexes"]
}
```

**Why partial index** — ลบสินค้าด้วย `deletedAt = now()` แต่ยังสร้างชื่อซ้ำได้สำหรับสินค้าใหม่ (เพราะ unique บังคับเฉพาะแถวที่ `deletedAt IS NULL`)

## 5. Prisma client extension — auto soft-delete filter

[lib/prisma.ts](../../lib/prisma.ts) ไม่ใช่แค่ singleton — ห่อ `PrismaClient` ด้วย `$extends({ name: "softDelete", query: {...} })` เพื่อ auto-inject `deletedAt: null` ให้ทุก read query ของ model ที่ทำ soft delete

```ts
const SOFT_DELETE_MODELS = new Set(["Product"]);
const FILTERED_READS = new Set([
  "findFirst", "findFirstOrThrow", "findMany",
  "count", "aggregate", "groupBy",
]);

function withSoftDelete(client) {
  return client.$extends({
    query: {
      $allModels: {
        async $allOperations({ model, operation, args, query }) {
          if (!SOFT_DELETE_MODELS.has(model)) return query(args);

          if (FILTERED_READS.has(operation)) {
            // inject deletedAt: null ถ้า caller ไม่ได้ระบุเอง
            if (!args.where || !("deletedAt" in args.where)) {
              args.where = { ...args.where, deletedAt: null };
            }
            return query(args);
          }

          if (operation === "findUnique" || operation === "findUniqueOrThrow") {
            // findUnique รับเฉพาะ unique field ใน where → inject ไม่ได้
            // → post-filter ผลลัพธ์เดี่ยว
            const row = await query(args);
            return row?.deletedAt == null ? row : null;
          }

          return query(args);
        },
      },
    },
  });
}
```

### Key points

| เรื่อง | Why |
|---|---|
| รวมศูนย์ที่ extension | caller เรียก `prisma.product.findMany()` ลืม `where: { deletedAt: null }` ไม่ได้ — extension ใส่ให้ |
| Opt-out ด้วย explicit `deletedAt` | admin "view deleted" page ส่ง `where: { deletedAt: { not: null } }` ได้ |
| `findUnique` ต้อง post-filter | Prisma บังคับ `where` ของ unique ใช้ unique field เท่านั้น → inject `deletedAt` ใน where ไม่ได้ → query ปกติแล้วเช็คผลลัพธ์ |
| `$allModels` + Set whitelist | ตอนนี้แค่ `Product`; เพิ่ม model อื่นในอนาคต = เพิ่มชื่อใน Set |

### ที่ extension **ไม่** ทำให้

- **Write operations** (`create`, `update`, `delete`) — ไม่แตะ → ลบจริงต้องเปลี่ยนเป็น `update({ data: { deletedAt: new Date() } })` เอง ที่ API layer
- **Raw query** (`$queryRaw`, `$executeRaw`) — bypass extension เต็มๆ
- **Relations** — ถ้า `User.orders` → `Order.items` → `Product`, ตัว product nested ไม่โดน filter (extension hit เฉพาะ top-level model ของ operation)
