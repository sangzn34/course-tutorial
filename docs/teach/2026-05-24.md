## บทที่ 3 — เริ่มทำ Login API (ยังไม่จบ — และมีจุดอันตราย)

### concept

API route ใน Next.js App Router = ไฟล์ `route.ts` ใน `app/api/.../`. export ฟังก์ชันชื่อ HTTP method (`POST`, `GET`) = endpoint นั้น

วันนี้เริ่ม:
- [app/api/auth/login/route.ts](../../app/api/auth/login/route.ts) — `POST` รับ username/password, validate ด้วย Zod, query `prisma.user.findUnique`
- `logout` / `me` — ยังเป็นไฟล์เปล่า (stub)

### 🔴 จุดอันตราย ต้องตามแก้ก่อนใช้จริง (สอนเป็นตัวอย่างของ "อย่าทำแบบนี้")

**1. รหัสผ่านเก็บ/เทียบแบบ plaintext** — โค้ดตอนนี้ query `where: { email, passwordHash: password }` คือเอารหัสที่พิมพ์มาเทียบตรง ๆ. **ผิดหลักความปลอดภัยร้ายแรง** — ต้อง:
- ตอนสมัคร: เก็บ `bcrypt.hash(password)` ไม่ใช่รหัสดิบ
- ตอน login: ดึง user มาแล้ว `bcrypt.compare(password, user.passwordHash)`

**2. รหัสหลุดใน log** — มี `console.log("Received login data:", { username, password })` → รหัสผ่านโผล่ใน log server. ลบทิ้งทันที

**3. ยังไม่มี session** — `logout`/`me` ว่าง = login แล้วระบบยังไม่ "จำ" ว่าใครล็อกอิน. ต้องเลือกวิธีเก็บ session (เช่น JWT ใน cookie, iron-session, NextAuth) ก่อนทำต่อ

### ⭐ takeaway
- **ห้ามเก็บรหัสผ่านเป็น plaintext** — hash ด้วย bcrypt เสมอ, เทียบด้วย `bcrypt.compare`
- **อย่า `console.log` ข้อมูลอ่อนไหว** (รหัสผ่าน, token) — มันค้างใน log
- login (ยืนยันตัวตน) กับ session (จำว่าใคร login) เป็นคนละเรื่อง — ต้องทำทั้งคู่

## บทที่ 4 — Soft Delete: "ลบ" แบบไม่ลบจริง

### concept

**Soft delete** = ไม่ลบแถวออกจาก DB จริง แต่ทำ mark ว่า "ถูกลบแล้ว" ด้วยคอลัมน์ `deletedAt` (เวลาที่ลบ; ถ้ายังไม่ลบ = `null`)

ทำไม? — กู้คืนได้, เก็บประวัติ, ไม่พังข้อมูลที่อ้างถึงมัน (เช่นออเดอร์เก่าที่อ้างสินค้านี้)

### โครงสร้าง schema — [prisma/schema.prisma](../../prisma/schema.prisma)

```prisma
model Product {
  name      String
  deletedAt DateTime?                          // null = ยังอยู่, มีค่า = ถูกลบ
  @@unique([name], where: { deletedAt: null }) // ชื่อห้ามซ้ำ "เฉพาะของที่ยังไม่ลบ"
}
```

`@@unique([name], where: { deletedAt: null })` คือ **partial unique index** — บังคับชื่อห้ามซ้ำ **เฉพาะแถวที่ยังไม่ถูกลบ**. ทำให้: ลบสินค้าชื่อ "ลาเต้" ไปแล้ว ยังสร้าง "ลาเต้" ใหม่ได้ (เพราะตัวเก่ามี `deletedAt` แล้ว ไม่นับ)

ต้องเปิด preview feature ของ Prisma:
```prisma
generator client { previewFeatures = ["partialIndexes"] }
```
> ⚠️ Prisma 7: `migrate dev` ของ partial index ยังต้อง interactive — ถ้า error ตอนรันใน CI/non-interactive ให้ใช้ `prisma migrate diff` + `migrate deploy` แทน

### ⭐ takeaway
- ข้อมูลสำคัญ (สินค้า, user, ออเดอร์) มักใช้ soft delete (`deletedAt`) ไม่ลบจริง
- partial unique index = บังคับ unique แบบมีเงื่อนไข (เช่น "unique เฉพาะที่ยังไม่ลบ")

## บทที่ 5 — ทำ Soft Delete ให้อัตโนมัติด้วย Prisma Extension

### concept

มี `deletedAt` แล้ว แต่ทุก query ต้องเขียน `where: { deletedAt: null }` เองทุกครั้ง — **ลืมเมื่อไหร่ = ของที่ลบแล้วโผล่กลับมา** เป็นบั๊ก/ช่องโหว่

วิธีแก้: ใช้ **Prisma Client Extension** — ห่อ Prisma ให้ **เติม `deletedAt: null` ให้ทุก read query อัตโนมัติ** caller ไม่ต้องจำ

### โค้ด — [lib/prisma.ts](../../lib/prisma.ts)

```ts
const SOFT_DELETE_MODELS = new Set(["Product"]);        // model ที่ทำ soft delete
const FILTERED_READS = new Set([                        // operation ที่ต้องกรอง
  "findFirst", "findFirstOrThrow", "findMany", "count", "aggregate", "groupBy",
]);

client.$extends({
  name: "softDelete",
  query: { $allModels: {
    async $allOperations({ model, operation, args, query }) {
      if (!SOFT_DELETE_MODELS.has(model)) return query(args);   // model อื่น ไม่ยุ่ง

      if (FILTERED_READS.has(operation)) {
        if (!args.where || !("deletedAt" in args.where))        // ถ้า caller ไม่ได้ระบุเอง
          args.where = { ...args.where, deletedAt: null };      // เติมให้
        return query(args);
      }

      if (operation === "findUnique" || operation === "findUniqueOrThrow") {
        const row = await query(args);                          // query ปกติก่อน
        return row?.deletedAt == null ? row : null;             // แล้วค่อยกรองผล
      }
      return query(args);
    },
  }},
});
```

ทำไม `findUnique` ต้องทำแยก? — เพราะ `findUnique` รับใน `where` **ได้แค่ field ที่เป็น unique** (เช่น `id`) จะยัด `deletedAt` เข้าไปไม่ได้ → เลย query ปกติแล้วค่อยเช็คผลลัพธ์ทีหลัง

### ⚠️ extension นี้ "ไม่" ทำให้ (ต้องจัดการเอง)
- **การลบจริง** (`delete`) — extension ไม่แตะ write. การ "ลบ" ต้องเปลี่ยนเป็น `update({ data: { deletedAt: new Date() } })` เองที่ API
- **`$queryRaw`** (SQL ดิบ) — ข้าม extension ทั้งหมด
- **ข้อมูลที่ join มา (relation)** — กรองเฉพาะ model ที่เป็นตัวหลักของ query เท่านั้น ตัวที่ join มาไม่โดน

### ⭐ takeaway
- กฎที่ "ต้องจำทุกครั้ง" = จุดที่จะลืม → ย้ายไปทำอัตโนมัติที่ส่วนกลาง (เช่น Prisma extension) ดีกว่าพึ่งความจำ
- เปิดช่อง opt-out ไว้ด้วย — เช่น admin อยากดูของที่ลบแล้ว ก็ส่ง `where: { deletedAt: { not: null } }` มาเอง extension จะไม่ทับ
