## บทที่ 1 — แยก schema ไว้ที่เดียว ใช้ได้ทั้ง client และ server

### concept

เดิม `loginScheme` ประกาศอยู่ในไฟล์ API route — ใช้ได้แค่ฝั่ง server. แต่ฟอร์ม login ฝั่ง browser ก็อยาก validate แบบเดียวกัน (เช็คก่อนส่ง ไม่ต้องรอ server ตอบ). ถ้าเขียน 2 ที่ → วันหนึ่งแก้ที่เดียวลืมอีกที่ = กติกาไม่ตรงกัน

ทางแก้: ย้าย schema ไปไฟล์กลาง [lib/schemas/auth.ts](../../lib/schemas/auth.ts) แล้ว import ใช้ทั้งสองฝั่ง — **single source of truth**

### โค้ด

```ts
// lib/schemas/auth.ts
export const loginSchema = z.object({
  email: z.string().trim().toLowerCase().email(),
  password: z.string().trim().min(8),
});

export type LoginInput = z.infer<typeof loginSchema>;
```

จุดน่าเรียน:

- **`.trim().toLowerCase()`** — zod ไม่ได้แค่ตรวจ แต่ **แปลงค่า (transform)** ให้ด้วย: ตัดช่องว่างหัวท้าย + บังคับอีเมลตัวเล็ก. ผู้ใช้พิมพ์ `" Sang@Mail.com "` → เข้าระบบเป็น `"sang@mail.com"` เสมอ → email ใน DB ไม่มีตัวซ้ำที่ต่างแค่ตัวพิมพ์
- **`z.infer<typeof loginSchema>`** — ได้ TypeScript type จาก schema อัตโนมัติ (`{ email: string; password: string }`) ไม่ต้องประกาศ type ซ้ำอีกรอบ. แก้ schema → type ตามให้เอง
- เปลี่ยนจาก `username` → `email` ทั้งระบบ (DB ใช้ email เป็น unique key อยู่แล้ว)
- มี `registerSchema` เตรียมไว้สำหรับหน้าสมัครสมาชิก (ยังไม่ได้ทำหน้า)

### ⭐ takeaway

- กติกา validate เขียน **ที่เดียว** ใน `lib/schemas/` แล้ว import ใช้ทั้งฟอร์ม (client) และ API (server)
- server ต้อง validate เสมอแม้ client เช็คแล้ว — client ข้ามได้ (ยิง API ตรง ๆ ได้เสมอ), client validate มีไว้เพื่อ UX
- zod ทำได้ทั้ง "ตรวจ" และ "แปลงค่า" — ใช้ normalize ข้อมูล (trim/lowercase) ตั้งแต่ขาเข้า
- `z.infer` = type ฟรีจาก schema, อย่าประกาศ interface ซ้ำเอง

## บทที่ 2 — หน้า `/login` ด้วย react-hook-form + zodResolver

### concept

ฟอร์มเขียนเองด้วย `useState` ทีละ field ได้ แต่พอมี validation + error message + สถานะกำลังส่ง จะรกเร็วมาก. **react-hook-form** จัดการให้หมด: เก็บค่า field, validate, error per-field, สถานะ `isSubmitting`. ต่อกับ zod ผ่าน **`zodResolver`** → ฟอร์มใช้ `loginSchema` ตัวเดียวกับ server เป๊ะ

### โค้ด — [app/login/page.tsx](../../app/login/page.tsx)

**1. ตั้งฟอร์ม:**

```tsx
const { register, handleSubmit, formState: { errors, isSubmitting } } =
  useForm<LoginInput>({ resolver: zodResolver(loginSchema) });
```

- `resolver: zodResolver(loginSchema)` = ให้ zod เป็นคนตัดสินว่าฟอร์ม valid ไหม (กติกาเดียวกับ API)
- `register("email")` = ผูก input เข้ากับฟอร์ม — กระจาย `{...register("email")}` ลงใน `<Input>` พอ ไม่ต้องเขียน `value`/`onChange` เอง
- `handleSubmit(onSubmit)` = ครอบฟังก์ชันเรา: validate ก่อน, ผ่านแล้วค่อยเรียก `onSubmit(data)` พร้อมข้อมูลที่ผ่านการตรวจ/แปลงแล้ว
- `isSubmitting` = จริงระหว่าง `onSubmit` ยังไม่จบ → ใช้ disable ปุ่ม + เปลี่ยนข้อความเป็น "กำลังเข้าสู่ระบบ..." กันกดรัว

**2. โชว์ error ใต้ field ด้วย component `Field` ของ shadcn:**

```tsx
<Field data-invalid={errors.email ? "true" : undefined}>
  <FieldLabel htmlFor="email">Email</FieldLabel>
  <Input id="email" type="email" autoComplete="email"
    aria-invalid={errors.email ? "true" : undefined}
    {...register("email")} />
  <FieldError errors={errors.email ? [errors.email] : undefined} />
</Field>
```

- `errors.email` มีค่าเฉพาะตอน field นั้นไม่ผ่าน → ใช้ทั้งโชว์ข้อความ (`FieldError`) และทำขอบแดง (`data-invalid`/`aria-invalid`)
- `aria-invalid` = บอก screen reader ว่า field นี้ผิด (accessibility) — ใส่คู่กับ visual เสมอ
- `autoComplete="email"` / `"current-password"` = บอก browser ให้เติมค่าที่จำไว้ + password manager ทำงานถูก

**3. ยิง API ด้วย `useMutation`** (ฝั่ง "เขียน" ของ React Query — คู่กับ `useQuery` ที่ไว้ "อ่าน"):

```tsx
const { mutateAsync } = useMutation({
  mutationFn: async (data: LoginInput) =>
    fetch("/api/auth/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(data),
    }).then(async (res) => {
      const body = await res.json();
      if (!res.ok) throw new Error(body?.message ?? "Login failed");
      return body;
    }),
});

const onSubmit = async (data: LoginInput) => {
  try {
    await mutateAsync(data);
    toast.success("เข้าสู่ระบบสำเร็จ!");
    router.push(redirect);
  } catch (error) {
    toast.error("เข้าสู่ระบบไม่สำเร็จ!");
  }
};
```

- `fetch` **ไม่ throw เองตอนได้ 401/422** — สำเร็จในสายตา fetch คือ "ได้ response" → ต้องเช็ค `res.ok` แล้ว throw เอง ไม่งั้น error จะกลายเป็น "สำเร็จ"
- `mutateAsync` = เวอร์ชัน Promise ของ mutation → `await` แล้วใช้ try/catch แยกทาง สำเร็จ/พลาด ได้ตรง ๆ
- `toast` (sonner) = แจ้งผลแบบป๊อปมุมจอ ไม่ต้องเขียน state โชว์ข้อความเอง

**4. จำไว้ว่าผู้ใช้กำลังจะไปไหน — query param `redirect`:**

```tsx
const searchParams = useSearchParams();
const redirect = searchParams.get("redirect") || "/menu";
// ...login สำเร็จ:
router.push(redirect);
// ...ลิงก์ไปสมัครก็พกต่อ:
<Link href={`/register?redirect=${encodeURIComponent(redirect)}`}>
```

แพตเทิร์นมาตรฐาน: ถูกเด้งมาหน้า login จากหน้าไหน (`/login?redirect=/checkout`) → login เสร็จส่งกลับหน้านั้น ไม่ใช่หน้าแรกเสมอ. `encodeURIComponent` กันค่า redirect มีอักขระพิเศษแล้ว URL พัง

### ⭐ takeaway

- ฟอร์มจริงจัง = `react-hook-form` + `zodResolver(schemaกลาง)` — ไม่เขียน `useState` ต่อ field
- อ่านข้อมูล → `useQuery`, ส่งข้อมูล/แก้ข้อมูล → `useMutation`
- `fetch` ต้องเช็ค `res.ok` เองเสมอ — 401 ไม่ใช่ exception ในสายตา fetch
- `isSubmitting` → disable ปุ่ม กัน submit ซ้ำ
- พกปลายทางผ่าน `?redirect=` แล้ว `router.push(redirect)` หลังสำเร็จ

## บทที่ 3 — เก็บรหัสผ่านแบบ hash ด้วย bcrypt

### concept

**ห้ามเก็บรหัสผ่านตรง ๆ ใน DB เด็ดขาด** — ถ้า DB หลุด รหัสผู้ใช้หลุดหมด (และคนชอบใช้รหัสซ้ำกับเว็บอื่น). ทางแก้: เก็บ **hash** = ผลจากฟังก์ชันทางเดียว — แปลงไปได้ แต่ย้อนกลับเป็นรหัสเดิมไม่ได้

ตอน login เราไม่ "ถอดรหัส" — เรา hash รหัสที่ผู้ใช้เพิ่งพิมพ์ แล้ว**เทียบ hash กับ hash**

bcrypt เหมาะกับงานนี้เพราะ:
- **ช้าโดยตั้งใจ** (ต่างจาก SHA ที่เร็ว) → เดารหัสแบบ brute force ทีละล้านครั้งทำได้ยาก
- มี **salt** ในตัว — สุ่มค่าผสมก่อน hash ทุกครั้ง → รหัสเดียวกัน hash ออกมาไม่เหมือนกัน กันตารางเทียบสำเร็จรูป (rainbow table)

### โค้ด — [lib/auth/password.ts](../../lib/auth/password.ts)

```ts
import bcrypt from "bcryptjs";

const COST = 12;

export function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, COST);
}

export function verifyPassword(password: string, hash: string): Promise<boolean> {
  return bcrypt.compare(password, hash);
}
```

- `COST = 12` = ความช้า: ยิ่งสูงยิ่งช้าแบบทวีคูณ (2¹² รอบ). 12 คือค่ามาตรฐานปัจจุบัน — ช้าพอกัน brute force, เร็วพอผู้ใช้ไม่รอนาน
- เทียบด้วย `bcrypt.compare(รหัสที่พิมพ์, hashใน DB)` เสมอ — **อย่า** hash เองแล้วเทียบ string (salt ทำให้ hash ใหม่ไม่ตรงกับของเก่าอยู่แล้ว และ compare ของ bcrypt เทียบแบบกัน timing attack)
- ใช้ `bcryptjs` (JS ล้วน) — ลงง่าย ไม่ต้อง compile native module

### ⭐ takeaway

- DB เก็บ `passwordHash` เท่านั้น — เห็นรหัส plaintext ใน DB/log ที่ไหน = บั๊กความปลอดภัย
- เทียบรหัสผ่าน = `bcrypt.compare()` เสมอ ไม่ใช่ `===`
- เลือกของที่ "ช้าโดยตั้งใจ" (bcrypt/argon2) สำหรับรหัสผ่าน — hash เร็ว ๆ (SHA/MD5) ไว้ทำอย่างอื่น

## บทที่ 4 — Session ด้วย JWT ใน httpOnly Cookie

### concept

HTTP ไม่มีความจำ — login เสร็จ request ถัดไป server ก็ไม่รู้ว่าเราเป็นใคร. ต้องมี **session**: หลักฐานที่ browser แนบมากับทุก request

วิธีที่ใช้วันนี้: server สร้าง **JWT** (JSON Web Token — ก้อนข้อมูล + ลายเซ็นดิจิทัล) แล้วฝากไว้ใน **cookie** ของ browser

- JWT ประกอบด้วย payload (userId, email, role) + **ลายเซ็น** ที่เซ็นด้วย secret ของ server → ใครแก้ payload เอง ลายเซ็นจะไม่ตรง server จับได้
- ⚠️ JWT **ไม่ได้เข้ารหัส** — แค่เซ็นกันแก้. ใครได้ token ไปก็อ่านข้างในได้ (ลอง paste ใน jwt.io ดู) → อย่าใส่ของลับใน payload

### โค้ด — [lib/auth/session.ts](../../lib/auth/session.ts)

**สร้าง token ด้วย `jose`:**

```ts
return new SignJWT({ sub: payload.userId, email: payload.email, role: payload.role })
  .setProtectedHeader({ alg: "HS256" })
  .setIssuedAt()
  .setExpirationTime("2h")
  .sign(new TextEncoder().encode(SECRET));
```

- `sub` (subject) = ช่องมาตรฐานของ JWT สำหรับ "token นี้ของใคร" — ใส่ userId
- `setExpirationTime("2h")` = token หมดอายุใน 2 ชั่วโมง → ถึง token หลุดก็ใช้ได้จำกัดเวลา
- ใช้ lib `jose` เพราะรันได้ทั้ง Node และ Edge runtime (middleware ของ Next รันบน Edge — เผื่ออนาคตต้อง verify ใน middleware)

**ฝากใน cookie:**

```ts
const store = await cookies();          // next/headers — ฝั่ง server เท่านั้น
store.set({
  name: "session",
  value: token,
  httpOnly: true,                        // JS ในหน้าเว็บอ่านไม่ได้
  secure: process.env.NODE_ENV === "production",  // prod ส่งผ่าน https เท่านั้น
  sameSite: "lax",                       // เว็บอื่นยิง request ข้ามมาไม่แนบ cookie (กัน CSRF ระดับหนึ่ง)
  path: "/",
  maxAge: 60 * 60 * 2,                   // อายุ cookie 2 ชม. (ให้ตรงกับอายุ token)
});
```

- **`httpOnly: true` คือหัวใจ** — cookie นี้ `document.cookie` อ่านไม่เห็น → ต่อให้โดน XSS (สคริปต์แปลกปลอมในหน้า) ก็ขโมย token ไม่ได้. นี่คือเหตุผลที่เก็บใน cookie ไม่เก็บใน `localStorage`
- `cookies()` ใน Next 16 เป็น **async** — ต้อง `await` (โค้ดเก่าตามเน็ตเป็น sync ก๊อปมาตรง ๆ จะพัง)
- browser แนบ cookie มากับทุก request ที่เข้า domain เราอัตโนมัติ — ฝั่ง client ไม่ต้องเขียนอะไรเพิ่มเลย (สังเกตว่าฟอร์ม login ไม่มีโค้ดจัดการ token สักบรรทัด)

**ฝั่ง API route** ([app/api/auth/login/route.ts](../../app/api/auth/login/route.ts)) — หา user จาก email แล้วตั้ง cookie:

```ts
const user = await prisma.user.findUnique({ where: { email } });
if (!user) {
  return Response.json({ message: "Invalid email or password" }, { status: 401 });
}
await setSessionCookie({ userId: user.id, email: user.email, role: user.role });
return Response.json({ message: "Login successful" });
```

- ข้อความ error เป็น "Invalid **email or password**" แบบกำกวมโดยตั้งใจ — ไม่บอกว่าผิดอันไหน กันคนนั่งไล่เช็คว่าอีเมลไหนมีบัญชี (user enumeration)
- ใส่ `role` ลง session ตั้งแต่ตอนนี้ → อนาคตทำหน้า admin เช็คจาก session ได้เลย

### ⭐ takeaway

- session token เก็บใน **httpOnly cookie** ไม่ใช่ `localStorage` — กัน XSS ขโมย
- JWT = เซ็นกันแก้ ไม่ใช่เข้ารหัส — อย่าใส่ความลับใน payload และต้องตั้งวันหมดอายุ
- error ของ login ต้องกำกวม ("email หรือ password ผิด") เสมอ
- `cookies()` ใน Next 16 ต้อง `await`

## 🔴 จุดอันตรายที่ยังค้าง — ต้องแก้ก่อนใช้จริง

โค้ดวันนี้เป็น **สถานะกลางบทเรียน** ยังมีช่องโหว่ที่รู้ตัวอยู่ 3 จุด:

1. **ยังไม่ได้เช็ครหัสผ่าน!** — ใน route ตอนนี้บรรทัด `verifyPassword` ถูก comment ไว้ (รอ seed user ใน DB ให้เป็น hash ก่อน) → ตอนนี้**รู้แค่ email ก็ login ได้**. เปิดใช้บล็อกนี้ให้เร็วที่สุด:
   ```ts
   if (!user || !(await verifyPassword(password, user.passwordHash))) {
     return Response.json({ message: "Invalid email or password" }, { status: 401 });
   }
   ```
   (สังเกตว่าเช็ค `!user` กับรหัสผิด **รวมเป็นเงื่อนไขเดียว ตอบ 401 เหมือนกัน** — เนียนกว่าแยกตอบ)
2. **secret ของ JWT hardcode อยู่ในไฟล์** [lib/auth/session.ts](../../lib/auth/session.ts) — ต้องย้ายไป `.env` (`SESSION_SECRET=...`) แล้วอ่านผ่าน `process.env`. และเพราะค่านี้ถูก commit ลง git แล้ว ถือว่า**หลุดถาวร** (อยู่ใน history) → ตอนย้ายต้อง**สุ่มค่าใหม่** ห้ามใช้ค่าเดิม
3. **`console.log` รหัสผ่าน** — `console.log("Received login data", { email, password })` ใน route = รหัสผ่าน plaintext โผล่ใน log ของ server. ลบทิ้ง (log password ไม่ได้ทุกกรณี ไม่ว่า dev หรือ prod)

> 📝 เทียบกับบันทึก [2026-05-24](./2026-05-24.md): จุดอันตราย "เทียบ plaintext ผ่าน `findUnique({ where: { email, passwordHash: password } })`" **แก้แล้ววันนี้** — เปลี่ยนเป็นหา user จาก email อย่างเดียว แล้ว(จะ)เทียบรหัสด้วย `bcrypt.compare`. แต่เกิดช่องโหว่ใหม่ชั่วคราวตามข้อ 1 ข้างบนแทน
